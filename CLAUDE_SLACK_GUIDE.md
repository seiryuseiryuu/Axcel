## 🔰 基本の使い方

### Claude にできること

| できること | 具体例 |
|-----------|--------|
| コード実装 | 新ツールの作成、バグ修正、リファクタリング |
| コード分析 | 既存コードの説明、影響範囲の調査 |
| 要件定義の読解 | 要件定義書を読んで実装方針を提案 |
| デバッグ | エラーログを渡して原因特定・修正 |
| デプロイ | Git push + Vercel デプロイまで実行 |
| ドキュメント作成 | README、技術仕様書、コメント追加 |

## 📝 Slack での指示の出し方

### ✅ 良い指示の例

```
🟢 具体的で明確な指示

「VSLライティングツールを新要件定義書に基づいて実装して。
 修正前にどのファイルをいじるか一覧を見せて。
 完了後 Vercel にデプロイして。」
```

```
🟢 エラー報告 + 対処依頼

「動画切り抜きツールで以下のエラーが出る。修正してデプロイして。
 エラー: TypeError: Cannot read property 'map' of undefined
 発生箇所: VideoClipWorkflow.tsx の STEP3
 再現手順: URL入力 → 分析ボタン → エラー」
```

```
🟢 修正範囲を限定した指示

「StudioSidebar.tsx のツール順番を以下に変更して:
 1. SEO記事
 2. YouTube台本
 3. VSLライティング
 4. 動画切り抜き
 他のファイルは触らないで。」
```

### ❌ 避けるべき指示

```
🔴 曖昧すぎる
「なんかバグってる、直して」
→ どの画面？どんなエラー？再現手順は？

🔴 範囲が広すぎる
「全ツールをリファクタして」
→ 具体的にどのツール？何をどう改善？

🔴 前提情報がない
「前の続きやって」
→ 何の続き？前回のコンテキストを簡潔に共有して
```

## 📋 指示テンプレート集

### 1️⃣ 新規ツール実装

```
📋 *新規ツール実装依頼*
———
*ツール名:* ○○ツール
*要件定義書:* 新要件定義書/○○ツール要件定義
*優先度:* 高 / 中 / 低
*特記事項:* （あれば）

👉 実装前にファイル一覧と変更計画を見せて
👉 完了後 GitHub push + Vercel デプロイ
———
```

### 2️⃣ バグ修正

```
🐛 *バグ修正依頼*
———
*発生ツール:* ○○ツール
*発生画面:* STEP○ の ○○部分
*エラー内容:* （エラーメッセージ or スクショ）
*再現手順:*
1. ○○を入力
2. ○○ボタンを押す
3. エラーが出る

👉 修正 → テスト → デプロイまでお願い
———
```

### 3️⃣ UI/デザイン修正

```
🎨 *UI修正依頼*
———
*対象画面:* /admin/studio/○○
*修正内容:*
• ボタンの色を青→緑に
• フォントサイズを大きく
• ローディング表示を追加

*参考:* （スクショやURLがあれば）
———
```

### 4️⃣ デプロイのみ

```
🚀 *デプロイ依頼*
———
現在のコードを GitHub + Vercel にデプロイして。
ビルドエラーがあれば修正もお願い。
———
```

### 5️⃣ コード調査・質問

```
🔍 *コード調査依頼*
———
*調べたいこと:* ○○ツールの○○処理がどう動いているか
*目的:* 改修前の影響範囲を把握したい
*関連ファイル:* （分かれば）
———
```

## 🔄 Claude とのやり取りフロー

```
1. 指示を出す（上のテンプレートを使用）
        ↓
2. Claude が変更計画を提示
   • 対象ファイル一覧
   • 各ファイルの変更内容
        ↓
3. 確認して承認 or 修正依頼
   「OK、進めて」 or 「○○は変更しないで」
        ↓
4. Claude が実装
        ↓
5. Claude がビルド確認 → デプロイ
        ↓
6. 完了報告（Slack フォーマット）
   • 変更内容
   • デプロイ先URL
   • 確認事項
```

## 💡 効果的に使うコツ

### スピードアップのコツ

| コツ | 説明 |
|------|------|
| 要件定義書を先に共有 | 「新要件定義書/○○を読んで」と一言添えるだけで精度UP |
| エラーはログごと貼る | スクショやコンソールログがあると一発解決しやすい |
| 既存ツールを参考指定 | 「VSLWorkflow.tsx と同じパターンで」と言えば統一感が出る |
| 1指示1タスク | 複数タスクは分けて依頼した方が確実 |

### 修正指示のコツ

```
✅ GOOD: 「STEP2 の分析結果表示を、箇条書きからテーブル形式に変えて」
❌ BAD:  「見た目をもうちょっといい感じにして」
```

### 承認/却下の伝え方

```
✅ 承認:   「OK、進めて」「LGTM」「問題なし、実装して」
🔄 修正:   「ファイル3は不要。1,2,4だけ進めて」
❌ 却下:   「やめて。代わりに○○して」
```

## ⚠️ 注意事項

1. **環境変数は Slack に貼らない** — API キーやシークレットは直接共有しない
2. **大きな変更は段階的に** — 一度に10ファイル以上変える場合は分割依頼
3. **デプロイ前にビルド確認を指示** — 「ビルド通るか確認してからデプロイ」
4. **本番障害時は最小修正を指示** — 「リファクタなし、最小修正で直して」
